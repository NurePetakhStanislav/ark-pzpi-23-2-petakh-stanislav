1.2 Аналіз аналогів: Огляд існуючих систем чи рішень. Які недоліки вони мають? Як нова система покращить ситуацію?
1.3 Бізнес-цілі та критерії: На які цілі спрямований проєкт? Яких результатів має бути досягнуто після впровадження системи?
1.4 Потреби клієнтів та ринку: Які основні потреби користувачів задовольняє система?
1.5 Бізнес-ризики: Які ризики можуть вплинути на успіх проєкту? Які технічні, організаційні чи юридичні проблеми можуть виникнути?
2.1 Опис призначення системи: Що робить система? Яку цінність вона створює для користувачів?
2.2 Опис бізнес-логіки: Як працюватиме система з погляду бізнес-процесів. Які дії виконуватимуть користувачі?
3.1 Рамки первинного випуску: Які функції будуть реалізовані у першій версії системи?
3.2 Рамки наступних випусків: Які функції планується додати у майбутніх оновленнях?
3.3 Обмеження та винятки: Які дії або функції не реалізуються у межах цього проєкту? Які обмеження накладаються на використання системи?
4.3 Опис програмних засобів та інструментів: Які технології будуть використані? Системи управління версіями, мови програмування, середовища розробки тощо.

	3 ОПИС ВИКОНАНОЇ РОБОТИ
	1.1 Аналіз проблеми
	Незважаючи на швидшу автоматизацію лабораторних досліджень, найповільнішим етапом залишається визначення потенційних ризиків для пацієнтів. Це призводить до збільшення черг та затримки у виданні своєчасного лікування при ускладненнях. Запропонована система дозволяє прискорити обробку результатів та своєчасне визначення можливих ризиків для пацієнта.

	1.2 Аналіз аналогів
	Сучасні аналоги зазвичай характеризуються зручним інтерфейсом, наявністю мобільного застосунку, можуть підтримувати міжнародні мови та можливістю графічного відображення результатів аналізів. Однак вони обмежуються базовими порівняннями з нормою, підтримують обмежений набір лабораторних показників, надають мінімальну інформацію про ризики та методи лікування.

	1.3 Бізнес-цілі та критерії
	BO-1: Зменшити час на обробку результатів на 30%.
	BO-2: Підтримувати точність результату не менше ніж на 90%.
	BO-3: Зменшити черги пацієнтів в 30%.
	SC-1: Середній час між отриманням аналізів, не враховуючи тих, що довше години.
	SC-2: Рекомендації системи співпадають з оцінкою лікаря  90%.
	SC-3: Середня кількість пацієнтів в день.

	1.4 Потреби клієнтів та ринку
	1. Інтерфейс веб-сторінки та мобільного застосунку має бути інтуїтивно зрозумілим.
	2. Система має надавати точні дані на основі аналізів крові.
	3. Система має мати архів для попередніх аналізів.
	4. Система має повідомити про успішне виконання роботи.
	5. Дані пацієнтів мають бути захищені від можливості прямого перегляду.
	6. Збережені дані можна знайти за датою, віком чи назвою файлу.
	7. Інтерфейс веб-сторінки та мобільного застосунку має підтримувати візуалізацію складу крові.
	8. Система має підтримувати можливість надати аналізам задовільний чи незадовільний статус.

	1.5 Бізнес-ризики
	1. Можлива конкуренція серед існуючих систем.
	2. Можливі ризики при перших чи майбутніх запусках програми.
	3. Система має бути добре налагоджена, щоб не піддавати ризику людей.
	4. Час на якісне навчання системи визначати аналізи та потреба в більшій кількості даних для точності.
	5. Потребується вдосконалення методів зберігання даних пацієнтів без порушення вимог законодавства.
	2.1 Опис призначення системи
	Система створює список можливих захворювань на основі аналізу крові пацієнта. Спеціаліст може визначити точний діагноз та надати відповідне лікування. Також система може використовуватися для навчання молодих спеціалістів, демонструючи всі можливі взаємозв'язки між аналізами та можливими захворюваннями.

	2.2 Опис бізнес-логіки
	Лабораторія отримує аналізи крові пацієнта, визначає точний склад, надсилає дані у форматі json, після чого система перевіряє отримані дані. На їх основі вона показує відхилення від норми та ймовірні захворювання. Спеціаліст може на основі результатів визначити, чи дійсно їх трактування вірне та поставити потрібну реакцію.

	3.1 Рамки первинного випуску
	1. Реєстрація та авторизація.
	2. Можливість змінити ім’я та профільну фотографію.
	3. Підтримка автентифікації користувача як спеціаліста.
	4. Додавання документів через пристрій та через екран (доступний тільки на сайті).
	5. Виведення аналізів та можливих ризиків.
	6. Можливість підтвердити чи спростувати аналізи (за наявності підтвердження, що користувач є спеціалістом).
	7. Можливість переглянути попередні дані користувача.
	8. Наявне попередження про трактування даних, якщо користувач не зазначений як спеціаліст.
	9. Наявні методи модерації для перевірки імен та профільних фотографій користувачів.

	3.2 Рамки наступних випусків
	1. Додавання користувацької та групової підписки.
	2. Підтримка багатомовного інтерфейсу.
	3. Реєстрація через Google чи Microsoft.

	3.3 Обмеження та винятки
	1. База даних не має зберігати ПІБ пацієнта.
	2. Дані не можуть бути перейменовані, щоб запобігти їх збереження як ПІБ.
	3. Система приймає та працює тільки з файлами формату JSON не більше 10 МБ.
	4. Працювати з системою можуть тільки зареєстровані користувачі.
	5. Статус даних може вирішувати тільки користувач з ролью спеціаліста. Можливі зміни статусу є “Valid” та “Invalid”.
	6. Зміни імені та профільної фотографії проходять модерацію і можуть увійти в силу після затвердження адміністратором.

	4.3 Опис програмних засобів та інструментів
Мова програмування: C# — для обробки даних аналізів, виявлення відхилень та формування висновків. Kotlin — для розробки мобільного застосунку на Android.
	Формат даних: JSON — для передачі результатів аналізів між лабораторією, сервером та клієнтським інтерфейсом.
Серверна частина: ASP.NET — для обробки запитів та взаємодії з базою даних.
Система управління версіями: Git — для відстеження змін у коді та документації.
Середовище розробки: Visual Studio — для зручного написання та налагодження коду.
База даних: SQL — для зберігання результатів аналізів та нормальних значень показників.
Веб-інтерфейс: HTML/CSS/JavaScript для відображення результатів у браузері.
Мобільний інтерфейс: Kotlin — для взаємодії з сервером та відображення результатів на пристроях користувачів.
